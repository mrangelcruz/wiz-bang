# wiz_automation_rule
Resource for Wiz Automation Rules. This resource is available from version >= 1.3.238

Automation Rules are if-this-then-that statements that trigger Actions, typically when a new Issue is generated by a Control. Automation Rules will run based on when the logical criteria you have set have been met.

## Before you begin
The prerequisites are:

- Familiarize yourself with Automation Rules
- Configured Wiz Terraform provider and a Wiz service account with these permissions:
- - create:automation_rules
- - read:automation_rules
- - update:automation_rules
- - delete:automation_rules
- - admin:run_response_action

## Use Cases

See below for examples of configuring Automation Rules with different types of Actions using the Wiz Terraform provider.

You can customize the body of tickets by using the available template variables.

# Automation rule for notifications services
resource "wiz_automation_rule" "notifications_services_rule" {
  name           = "notification-services-automation-rule"
  trigger_source = "ISSUES"
  filters = jsonencode({
    "severity" : [
      "CRITICAL", "HIGH"
    ]
  })
  trigger_type = [
    "CREATED", "RESOLVED"
  ]
  description = "Automation rule that sends notifications to CSP's notification services"
  actions {
    integration_id = "743d527d-aacc-4154-a013-9f827733a869"
    action_type    = "AWS_SNS"
    params {
      aws_sns {
        body = var.message_body

  }
  actions {
    integration_id = "919d18df-cca6-44b2-a527-15ce32865d19"

## Schema
### Required
- filters (String) Each trigger source has its own selection of filters:
- - If TriggerSource is ISSUES, then filters should be of AutomationRuleTriggerSource type
- - If TriggerSource is CLOUD_EVENTS then filters should be of AutomationRuleCloudEventFilters type
- - The filters field must be a valid JSON format.
- name (String) Name of the Automation Rule.
- trigger_type (List of String) The trigger types of the Automation Rule. Possible values: CREATED, UPDATED, RESOLVED, REOPENED, DETECTED, DUE, STATUS_CHANGED.


## Optional
- actions (Block Set) Actions to execute once an Automation Rule event is triggered and passes the filters. Only relevant for integrations. (see below for nested schema)
- auto_remediation (Block Set, Max: 1) The auto-remediation to execute once an Automation Rule event is triggered and passes the filters. Only relevant for targeted auto-remediation integrations. (see below for nested schema)
- description (String) Description of the Automation Rule.
- enabled (Boolean) Determines if rule is enabled or disabled
- - Default: true.
- project (String) The project this rule is scoped to. Leaving it empty will scope the rule to all the projects. Changing this value will force a new rule due to Wiz API restrictions. If you set a value for this property, do not use the project filter in the filters field.
- trigger_parameters (Block List, Max: 1) The trigger parameters of the Automation Rule. Required when setting the trigger type to DUE. (see below for nested schema)
- trigger_source (String) The source that triggers the Automation Rule. Possible values: ISSUES, CLOUD_EVENTS, CONTROL, CONFIGURATION_FINDING, AUDIT_LOGS, SYSTEM_HEALTH_ISSUES, CI_CD_SCANS, DATA_FINDINGS, SYSTEM_ACTIVITIES, ADMISSION_REVIEWS, DETECTIONS, CLOUD_COST_OPPORTUNITIES, SERVICE_ISSUES, POSTURE_ISSUES, RUNTIME_EVENTS, THREATS, POLICY_UPDATES, CLOUD_COST_MONITOR_FINDINGS. Default: ISSUES.

## Read-Only
- created_at (String) Identifies the date and time when the object was created.
- created_by (String) The ID of the user who created/updated this rule.
- id (String) Unique identifier for the Automation Rule.
- last_execution_error (String) The last execution error this Automation Rule encountered.
- updated_at (String) Identifies the date and time when the object was updated.
